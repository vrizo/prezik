Recording

CDP is the Chrome DevTools Protocol: the wire protocol Chrome exposes so external tools can control and observe the browser. Playwright itself drives Chrome over CDP. One CDP command, Page.startScreencast, makes Chrome push a JPEG image of every rendered frame to us.

Capture: we run our own screencast instead of Playwright's recordVideo. recordVideo uses the same screencast internally but re-encodes to VP8 at a fixed low bitrate (~1 Mbit/s), which looks muddy. We take the JPEG frames at quality 80 with their timestamps, acknowledge each frame immediately (the disk write happens off the critical path so it never throttles Chrome's frame rate), and assemble them with ffmpeg at CRF 18 into h264 at a constant 30 fps. The page is laid out at the format's CSS viewport (1280x720 for horizontal 16:9, 720x1280 for vertical 9:16) at deviceScaleFactor 1.5 (elements read comfortably large) while Chrome renders at 1.5x device pixels (1920x1080 or 1080x1920), so the frames are native full HD — this needs both the Playwright context option and the --force-device-scale-factor launch flag, because the screencast emits DIP-sized frames unless the headless compositor itself runs at 1.5. ffmpeg is used for assembly and audio muxing only, never for zooming.

Zoom: in-page CSS transform on the html element (smooth scale to the target with padding). Decided after an empirical comparison against browser-level zoom and pinch emulation — see recorder/README.md for why playwright-zoom (whole-tab zoom, no coordinates, extension does not load headless) could not do this. The page's own inline root transform is saved before zoom and restored on zoom-out. Zooms chain: a zoom while already zoomed pans and rescales directly to the next element (translate + scale around a fixed origin) without passing through scale 1, so element-to-element moves don't flicker through a full zoom-out. The geometry is pure math in recorder/src/zoomMath.ts (unit-tested); browser.ts measures in the page, computes in Node, applies in the page.

Cursor and highlights: fixed-position elements injected into the page, so they scale with the zoom like everything else. While a zoom is active their coordinates are converted from visual (post-transform) to layout space, since fixed elements inside the transformed html are positioned in layout coordinates.

Audio: per-scene OpenAI TTS mp3s (fast, energetic delivery via voice instructions, then time-compressed ~10% with ffmpeg atempo since gpt-4o-mini-tts ignores the API speed parameter), durations measured with ffprobe, scenes padded so narration fits, tracks placed at scene offsets in the final mux. Captions (when enabled) are burned into the video by ffmpeg's subtitles filter from an SRT file, and also emitted as a sidecar WebVTT for the in-app player track — both built from the same narrations and offsets.

Pacing: a scene's leading goto runs before its narration clock starts, followed by a bounded networkidle wait, so page loads happen in the silent gap between narration clips instead of on camera. Within a scene, the visual beats (highlight/zoom/zoomOut) are spread evenly across the narration (timing.ts beatStartOffsets, min 1 s apart) so the viewer sees each element while the voice describes it and the video never freezes for long. A highlight's full fade-in/hold/fade-out is tracked on the Node side, and the recorder waits for it to finish before any goto, before the outro card, and before a scene can end — so the video never navigates or cuts mid-animation.

Silence compression: after capture, silent stretches of video — spans not covered by any narration clip, which is where the goto loads land — are cut down to at most 800 ms (screencast.ts compressSilence). Only silent video time is removed, keeping the last 800 ms of each gap so the held frame is the loaded page, never a stale spinner; later frame timestamps and clip offsets shift left together, so A/V sync is untouched. Silent renders (tts off) skip compression and keep real-time pacing.

Storage: the container hands the finished mp4 and VTT to its own Worker, which writes them to an R2 bucket binding. Playback is a plain HTML5 video tag with a captions track, served from R2 with range support. No video service subscription.
